# Controllers in ASP.NET Core Web API

Controllers are classes that handle incoming HTTP requests, process them, and return responses to clients. In ASP.NET Core Web API, controllers are the main entry point for API endpoints.

---

## Why Controllers are Needed

Without controllers, ASP.NET Core would not know:

* Which URL should invoke which method
* How to process HTTP requests
* How to return data in JSON format
* How to handle parameters sent by clients

Controllers act as the bridge between:

```
Client
   ↓
Controller
   ↓
Business Logic (Services/MediatR)
   ↓
Database
```

---

# Creating a Controller

A controller is simply a class that:

* Inherits from `ControllerBase`
* Is decorated with `[ApiController]`
* Has a `[Route()]` attribute

```csharp
using Microsoft.AspNetCore.Mvc;

namespace MyApi.Controllers
{
    [ApiController]
    [Route("api/[controller]")]
    public class ProductsController : ControllerBase
    {

    }
}
```

---

# Understanding the Attributes

## 1. `[ApiController]`

Enables API-specific features.

### Benefits

### Automatic Model Validation

Without `[ApiController]`

```csharp
if(!ModelState.IsValid)
{
    return BadRequest(ModelState);
}
```

With `[ApiController]`

```csharp
// No need to check ModelState
```

ASP.NET automatically returns:

```json
{
   "errors":{
      "Name":["The Name field is required."]
   }
}
```

---

### Automatic Parameter Binding

ASP.NET automatically determines where data comes from.

Example

```csharp
public IActionResult Get(int id)
```

ASP.NET understands that `id` comes from the URL.

---

## 2. `[Route()]`

Defines the URL pattern.

```csharp
[Route("api/[controller]")]
```

If controller name is:

```csharp
ProductsController
```

URL becomes:

```text
api/products
```

# HTTP Verbs

Controllers expose actions using HTTP methods.

| Verb   | Attribute      | Purpose                |
| ------ | -------------- | ---------------------- |
| GET    | `[HttpGet]`    | Retrieve data          |
| POST   | `[HttpPost]`   | Create data            |
| PUT    | `[HttpPut]`    | Update entire resource |
| PATCH  | `[HttpPatch]`  | Partial update         |
| DELETE | `[HttpDelete]` | Remove data            |

# Action Methods

Action methods are public methods inside a controller that handle requests.

```csharp
public IActionResult Get()
{
    return Ok();
}
```

Every action method returns an `IActionResult` or `ActionResult<T>`.

---

# Common Return Types

### Ok()

Returns **200 OK**

```csharp
return Ok(products);
```

---

### CreatedAtAction()

Returns **201 Created**

```csharp
return CreatedAtAction(
        nameof(Get),
        new { id = product.Id },
        product);
```

---

### BadRequest()

Returns **400 Bad Request**

```csharp
return BadRequest();
```

---

### Unauthorized()

Returns **401 Unauthorized**

```csharp
return Unauthorized();
```

---

### NotFound()

Returns **404 Not Found**

```csharp
return NotFound();
```

---

### NoContent()

Returns **204 No Content**

```csharp
return NoContent();
```

# Example Controller

```csharp
using Microsoft.AspNetCore.Mvc;

namespace MyApi.Controllers
{
    [ApiController]
    [Route("api/[controller]")]
    public class EmployeesController : ControllerBase
    {
        [HttpGet]
        public IActionResult GetAll()
        {
            return Ok("All Employees");
        }

        [HttpGet("{id}")]
        public IActionResult Get(int id)
        {
            return Ok($"Employee {id}");
        }

        [HttpPost]
        public IActionResult Create(EmployeeDto dto)
        {
            return Created("", dto);
        }

        [HttpDelete("{id}")]
        public IActionResult Delete(int id)
        {
            return NoContent();
        }
    }
}
```

# Controllers in Your SmartHealthcare Project

Since you're using **CQRS + MediatR**, your controllers should remain very thin and delegate work to handlers.

Example:

```csharp
[ApiController]
[Route("api/auth")]
public class AuthController : ControllerBase
{
    private readonly IMediator _mediator;

    public AuthController(IMediator mediator)
    {
        _mediator = mediator;
    }

    [HttpPost("login")]
    public async Task<IActionResult> Login(LoginCommand command)
    {
        var result = await _mediator.Send(command);

        return Ok(result);
    }

    [HttpGet("me")]
    public async Task<IActionResult> Me()
    {
        var result = await _mediator.Send(
                        new GetCurrentUserQuery());

        return Ok(result);
    }
}
```

This approach is common in enterprise applications because controllers are responsible only for:

* Accepting requests
* Sending Commands/Queries through MediatR
* Returning HTTP responses

All business logic stays inside handlers, making controllers easy to maintain and test.
