# Services in ASP.NET Core Web API

Services are classes that contain the **business logic** of an application. Instead of placing logic inside controllers, controllers delegate work to services.

Services help follow the **Separation of Concerns (SoC)** principle by keeping controllers thin and focused only on handling HTTP requests and responses.

---

# Why Services are Needed

Without services, controllers become large and difficult to maintain.

### Bad Practice

```csharp
[HttpPost]
public async Task<IActionResult> Register(RegisterRequest request)
{
    var user = new User
    {
        Name = request.Name,
        Email = request.Email
    };

    _context.Users.Add(user);
    await _context.SaveChangesAsync();

    await _emailService.SendEmailAsync(user.Email);

    return Ok();
}
```

Problems:

* Controller contains business logic
* Hard to test
* Difficult to reuse
* Violates Single Responsibility Principle

---

### Better Approach

```csharp
[HttpPost]
public async Task<IActionResult> Register(RegisterRequest request)
{
    await _userService.RegisterAsync(request);

    return Ok();
}
```

Business logic moves to a service.

---

# Creating a Service

## Step 1: Create an Interface

```csharp
public interface IUserService
{
    Task RegisterAsync(RegisterRequest request);
}
```

---

## Step 2: Implement the Interface

```csharp
public class UserService : IUserService
{
    private readonly ApplicationDbContext _context;

    public UserService(ApplicationDbContext context)
    {
        _context = context;
    }

    public async Task RegisterAsync(RegisterRequest request)
    {
        var user = new User
        {
            Name = request.Name,
            Email = request.Email
        };

        _context.Users.Add(user);

        await _context.SaveChangesAsync();
    }
}
```

---

## Step 3: Register the Service

In `Program.cs`

```csharp
builder.Services.AddScoped<IUserService, UserService>();
```

---

## Step 4: Inject the Service

```csharp
[ApiController]
[Route("api/users")]
public class UsersController : ControllerBase
{
    private readonly IUserService _userService;

    public UsersController(IUserService userService)
    {
        _userService = userService;
    }

    [HttpPost]
    public async Task<IActionResult> Register(RegisterRequest request)
    {
        await _userService.RegisterAsync(request);

        return Ok();
    }
}
```

# Service Lifetimes

ASP.NET Core provides three lifetimes for services.

| Lifetime  | Instance Created              | Typical Usage          |
| --------- | ----------------------------- | ---------------------- |
| Transient | Every request for service     | Lightweight services   |
| Scoped    | Once per HTTP request         | EF Core DbContext      |
| Singleton | Once for application lifetime | Caching, configuration |

---

## Transient

```csharp
builder.Services.AddTransient<IEmailService, EmailService>();
```

A new instance is created every time it is requested.

---

## Scoped

```csharp
builder.Services.AddScoped<IUserService, UserService>();
```

One instance is shared during a single HTTP request.

Example:

```text
Request 1
 ├── UserService Instance A
 └── DbContext Instance A

Request 2
 ├── UserService Instance B
 └── DbContext Instance B
```

---

## Singleton

```csharp
builder.Services.AddSingleton<ICacheService, CacheService>();
```

Only one instance exists throughout the application's lifetime.

# Types of Services

---

## Business Services

Contain application logic.

```csharp
public interface IAppointmentService
{
    Task BookAppointmentAsync();
}
```

---

## Infrastructure Services

Interact with external systems.

Examples:

* Email Service
* SMS Service
* File Storage Service
* Payment Gateway Service

Example:

```csharp
public interface IEmailService
{
    Task SendEmailAsync(
        string email,
        string subject,
        string body);
}
```

---

## Repository Services

Used for database operations.

```csharp
public interface IUserRepository
{
    Task<User?> GetByIdAsync(Guid id);
}
```

---

# Dependency Injection

ASP.NET Core has a built-in Dependency Injection container.

Without DI:

```csharp
var service = new UserService();
```

With DI:

```csharp
public UsersController(IUserService service)
{
    _service = service;
}
```

Advantages:

* Loose coupling
* Easier testing
* Better maintainability
* Service replacement becomes simple

# Example Email Service

Interface

```csharp
public interface IEmailService
{
    Task SendEmailAsync(
        string to,
        string subject,
        string body);
}
```

Implementation

```csharp
public class EmailService : IEmailService
{
    public async Task SendEmailAsync(
        string to,
        string subject,
        string body)
    {
        // SMTP Logic
    }
}
```

Registration

```csharp
builder.Services.AddScoped<IEmailService, EmailService>();
```

Usage

```csharp
await _emailService.SendEmailAsync(
    user.Email,
    "Welcome",
    "Registration successful");
```

# Services vs Repositories

| Feature                 | Service   | Repository |
| ----------------------- | --------- | ---------- |
| Contains business rules | ✔         | ✘          |
| Accesses database       | Sometimes | ✔          |
| Calls external APIs     | ✔         | ✘          |
| Validation logic        | ✔         | ✘          |
| Data persistence        | ✘         | ✔          |

# Services in a CQRS Architecture


```text
Client
   │
   ▼
Controller
   │
   ▼
MediatR
   │
   ▼
Command/Query Handler
   │
   ├── EmailService
   ├── JwtService
   ├── CurrentUserService
   └── DbContext
   │
   ▼
Database
```

In a CQRS-based application, **handlers often replace traditional business services**, while infrastructure-related operations are still implemented as services. This keeps controllers thin, business logic centralized in handlers, and integrations with external systems isolated in dedicated services.
