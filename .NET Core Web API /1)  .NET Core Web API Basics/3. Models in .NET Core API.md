# Models in ASP.NET Core Web API

Models are classes that represent the data used by your application. In ASP.NET Core Web API, models are used to:

* Transfer data between client and server
* Represent database entities
* Validate incoming requests
* Shape API responses

---

# Types of Models in ASP.NET Core Web API

There are generally four kinds of models used in enterprise applications.

## 1. Entity Models

Entity models represent tables in the database and are managed by **EF Core**.

Example:

```csharp
public class Patient
{
    public Guid Id { get; set; }

    public string FirstName { get; set; } = string.Empty;

    public string LastName { get; set; } = string.Empty;

    public DateTime DateOfBirth { get; set; }
}
```

Database table

| Id | FirstName | LastName | DateOfBirth |
| -- | --------- | -------- | ----------- |
| 1  | John      | Doe      | 1995-05-20  |

### Characteristics

* Stored in the Domain layer
* Mapped to database tables
* Used by EF Core migrations
* Usually contain navigation properties

Example:

```csharp
public class Appointment
{
    public Guid Id { get; set; }

    public Guid PatientId { get; set; }

    public Patient Patient { get; set; } = default!;
}
```

---

## 2. DTOs (Data Transfer Objects)

DTOs are used to exchange data between the API and clients.

They help avoid exposing database entities directly.

Example:

```csharp
public class RegisterPatientDto
{
    public string FirstName { get; set; } = string.Empty;

    public string LastName { get; set; } = string.Empty;

    public string Email { get; set; } = string.Empty;

    public string Password { get; set; } = string.Empty;
}
```

Controller

```csharp
[HttpPost]
public async Task<IActionResult> Register(
        RegisterPatientDto dto)
{
    return Ok();
}
```

### Why DTOs are important

Suppose your entity contains sensitive data.

```csharp
public class ApplicationUser
{
    public string Email { get; set; } = string.Empty;

    public string PasswordHash { get; set; } = string.Empty;

    public string SecurityStamp { get; set; } = string.Empty;
}
```

Returning this entity directly is unsafe.

Instead:

```csharp
public class UserResponseDto
{
    public string Email { get; set; } = string.Empty;
}
```

---

## 3. Request Models

Request models describe the data clients send to the API.

Example:

```csharp
public class LoginRequest
{
    public string Email { get; set; } = string.Empty;

    public string Password { get; set; } = string.Empty;
}
```

Request body

```json
{
  "email":"john@example.com",
  "password":"Password@123"
}
```

Controller

```csharp
[HttpPost]
public IActionResult Login(LoginRequest request)
{
    return Ok();
}
```

---

## 4. Response Models

Response models define what the API sends back.

Example:

```csharp
public class LoginResponse
{
    public string AccessToken { get; set; } = string.Empty;

    public string RefreshToken { get; set; } = string.Empty;

    public DateTime ExpiresAt { get; set; }
}
```

Response

```json
{
  "accessToken":"eyJ...",
  "refreshToken":"abc123",
  "expiresAt":"2026-06-19T15:30:00"
}
```

# Model Binding

ASP.NET Core automatically converts incoming data into model objects.

Example:

Request

```http
POST /api/auth/login
```

Body

```json
{
   "email":"john@example.com",
   "password":"Password@123"
}
```

Controller

```csharp
[HttpPost]
public IActionResult Login(LoginRequest request)
{
    return Ok();
}
```

ASP.NET Core creates the `LoginRequest` object automatically.

# Model Validation

Models can include validation attributes.

```csharp
using System.ComponentModel.DataAnnotations;

public class RegisterPatientDto
{
    [Required]
    public string FirstName { get; set; } = string.Empty;

    [Required]
    [EmailAddress]
    public string Email { get; set; } = string.Empty;

    [MinLength(8)]
    public string Password { get; set; } = string.Empty;
}
```

Invalid request

```json
{
   "email":"invalid-email"
}
```

Response

```json
{
  "errors": {
    "FirstName": [
      "The FirstName field is required."
    ],
    "Email": [
      "The Email field is not a valid e-mail address."
    ]
  }
}
```

With `[ApiController]`, this validation happens automatically.

# ViewModel vs DTO

| Feature                     | DTO | ViewModel |
| --------------------------- | --- | --------- |
| Used in Web API             | ✔   | Rarely    |
| Used in MVC Views           | ✘   | ✔         |
| Transfers Data              | ✔   | ✔         |
| Contains UI-specific fields | ✘   | ✔         |

Example ViewModel:

```csharp
public class PatientViewModel
{
    public string FullName { get; set; } = string.Empty;

    public int Age { get; set; }

    public bool CanEdit { get; set; }
}
```

# Models in Your SmartHealthcare Project

Based on your architecture, models are distributed across layers.

### Domain Layer

```text
SmartHealthcare.Domain
│
├── Entities

│   ├── Hospital.cs
│   ├── DoctorProfile.cs
│   ├── PatientProfile.cs
```

These are **Entity Models**.

### Application Layer

```text

│
├── Features
│   └── Auth
│       ├── Commands

│       └── DTOs
```

Examples:

```csharp

}
```

# Best Practices

✅ Use entities only for persistence.

✅ Never expose EF Core entities directly from controllers.

✅ Use DTOs for requests and responses.

✅ Add validation attributes or use **FluentValidation**.

✅ Keep models focused on a single responsibility.

For your **SmartHealthcare** project, the recommended flow is:

```text
Client Request
      │
      ▼
Request DTO / Command
      │
      ▼
MediatR Handler
      │
      ▼
Domain Entity
      │
      ▼
Database
      │
      ▼
Response DTO
      │
      ▼
Client
```

This separation is the pattern typically used in enterprise-grade ASP.NET Core Web APIs built with **CQRS, MediatR, and EF Core**.
