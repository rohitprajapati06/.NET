# Service Lifetimes in ASP.NET Core Web API

ASP.NET Core's Dependency Injection container supports **three service lifetimes**:

| Lifetime      | Instance Created                    | Shared Between Requests? | Typical Usage                        |
| ------------- | ----------------------------------- | ------------------------ | ------------------------------------ |
| **Transient** | Every time it is requested          | ❌ No                     | Lightweight stateless services       |
| **Scoped**    | Once per HTTP request               | ❌ No                     | Database contexts, business services |
| **Singleton** | Once for the application's lifetime | ✅ Yes                    | Caching, configuration, logging      |

---

## 1. Transient

A new object is created **every time** the service is requested.

### Registration

```csharp
services.AddTransient<IEmailService, EmailService>();
```

### Example

```csharp
public class EmailService : IEmailService
{
    public Guid Id { get; } = Guid.NewGuid();
}
```

```csharp
public class UserService
{
    public UserService(IEmailService service1,
                       IEmailService service2)
    {
        Console.WriteLine(service1.Id);
        Console.WriteLine(service2.Id);
    }
}
```

### Output

```text
a1b2c3
d4e5f6
```

Different IDs mean different instances.

### When to use

✅ Email service
✅ PDF generator
✅ Validation service
✅ Utility/helper classes

### Avoid

❌ Expensive object creation
❌ Database connections

---

## 2. Scoped

A single instance is created **for each HTTP request**.

### Registration

```csharp
services.AddScoped<IUserService, UserService>();
```

### Example

Request 1:

```text
Service Id = 1111
Service Id = 1111
```

Request 2:

```text
Service Id = 2222
Service Id = 2222
```

Within the same request, the instance is reused.

---

### Why is DbContext Scoped?

```csharp
services.AddDbContext<ApplicationDbContext>();
```

Internally, this registers it as **Scoped**.

Benefits:

* One database context per request
* Tracks entity changes consistently
* Prevents concurrency issues
* Maintains a single transaction boundary

### Typical Scoped Services

```csharp
services.AddScoped<IAppointmentService, AppointmentService>();
services.AddScoped<IDoctorService, DoctorService>();
services.AddScoped<IHospitalService, HospitalService>();
```

For your **Smart Healthcare System**, most business services should be Scoped.

Examples:

* `AppointmentService`
* `DoctorAvailabilityService`
* `AuthService`
* `HospitalImportService`

---

## 3. Singleton

Only one instance exists for the entire application lifetime.

### Registration

```csharp
services.AddSingleton<ICacheService, CacheService>();
```

### Example

Request 1

```text
Service Id = ABC
```

Request 2

```text
Service Id = ABC
```

Request 1000

```text
Service Id = ABC
```

Always the same object.

---

### Suitable Singleton Services

```csharp
services.AddSingleton<IMemoryCache, MemoryCache>();
services.AddSingleton<IJwtTokenGenerator, JwtTokenGenerator>();
```

Examples:

* In-memory cache
* Configuration providers
* Logging services
* Static lookup data

---

## Lifetime Hierarchy Rule

A longer-lived service **must not depend on** a shorter-lived service.

### Wrong

```csharp
services.AddSingleton<MyService>();
services.AddScoped<ApplicationDbContext>();
```

```csharp
public class MyService
{
    public MyService(ApplicationDbContext db)
    {
    }
}
```

This throws:

```text
Cannot consume scoped service
'ApplicationDbContext'
from singleton 'MyService'
```

---

### Correct

Singleton → Singleton ✅

Scoped → Scoped ✅

Scoped → Transient ✅

Transient → Scoped ✅

Singleton → Scoped ❌

Singleton → Transient ⚠️ (allowed, but the transient effectively behaves like a singleton because it is created only once during singleton construction)

---


### Rule of thumb

* **Transient** → Stateless helper services
* **Scoped** → Services that work with request data or the database
* **Singleton** → Shared application-wide services and caches

For enterprise ASP.NET Core APIs, **Scoped is usually the default choice for application and domain services**.
