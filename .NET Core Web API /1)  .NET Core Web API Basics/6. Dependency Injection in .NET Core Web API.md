# Dependency Injection (DI) in ASP.NET Core Web API

Dependency Injection (DI) is a design pattern used to achieve **Inversion of Control (IoC)** by allowing objects to receive their dependencies from an external source instead of creating them themselves.

ASP.NET Core has a built-in DI container.

---

# Why Dependency Injection?

Without DI:

```csharp
public class AppointmentController : ControllerBase
{
    private readonly AppointmentService _service;

    public AppointmentController()
    {
        _service = new AppointmentService();
    }
}
```

### Problems

❌ Tight coupling

❌ Difficult to unit test

❌ Changing implementation requires changing controller code

---

With DI:

```csharp
public class AppointmentController : ControllerBase
{
    private readonly IAppointmentService _service;

    public AppointmentController(IAppointmentService service)
    {
        _service = service;
    }
}
```

ASP.NET Core injects the implementation automatically.

Advantages:

✔ Loose coupling

✔ Better testability

✔ Easier maintenance

✔ Easy implementation replacement

---

# How DI Works in ASP.NET Core

ASP.NET Core follows three steps.

### Step 1: Create Interface

```csharp
public interface IAppointmentService
{
    Task<List<Appointment>> GetAllAsync();
}
```

---

### Step 2: Create Implementation

```csharp
public class AppointmentService : IAppointmentService
{
    public async Task<List<Appointment>> GetAllAsync()
    {
        return new List<Appointment>();
    }
}
```

---

### Step 3: Register Service

In `Program.cs`

```csharp
builder.Services.AddScoped<IAppointmentService,
                           AppointmentService>();
```

---

### Step 4: Inject Service

```csharp
[ApiController]
[Route("api/[controller]")]
public class AppointmentController : ControllerBase
{
    private readonly IAppointmentService _service;

    public AppointmentController(IAppointmentService service)
    {
        _service = service;
    }

    [HttpGet]
    public async Task<IActionResult> Get()
    {
        var data = await _service.GetAllAsync();

        return Ok(data);
    }
}
```

# Service Lifetimes

ASP.NET Core supports three lifetimes.

## 1. Transient

Creates a new instance every time it is requested.

```csharp
builder.Services.AddTransient<IEmailService,
                              EmailService>();
```

Example:

```csharp
Controller → Service A → Service B

Service A = Instance 1
Service B = Instance 2
```

Use for:

* Email sending
* PDF generation
* Stateless helpers

---

## 2. Scoped

One instance per HTTP request.

```csharp
builder.Services.AddScoped<IAppointmentService,
                           AppointmentService>();
```

Example:

```text
Request 1
 ├── Controller
 ├── Service
 └── DbContext

All share the same instance


Request 2
 ├── Controller
 ├── Service
 └── DbContext

New instance created
```

Use for:

* Business services
* Repository classes
* EF Core DbContext

This is the most commonly used lifetime in Web APIs.

---

## 3. Singleton

Only one instance for the entire application lifetime.

```csharp
builder.Services.AddSingleton<ICacheService,
                              CacheService>();
```

Example:

```text
Application Starts

CacheService Instance #1

Request 1 → Uses #1
Request 2 → Uses #1
Request 3 → Uses #1
```

Use for:

* Caching
* Configuration providers
* Logging providers

Avoid storing request-specific data in singletons.

---

# Constructor Injection

Most commonly used approach.

```csharp
public class DoctorService
{
    private readonly ApplicationDbContext _db;

    public DoctorService(ApplicationDbContext db)
    {
        _db = db;
    }
}
```

---

# Registering DbContext

```csharp
builder.Services.AddDbContext<ApplicationDbContext>(
    options =>
        options.UseSqlServer(
            builder.Configuration.GetConnectionString("DefaultConnection")));
```

`AddDbContext()` registers `ApplicationDbContext` as a **Scoped** service.

# Interview Questions

**Q1. What is Dependency Injection?**

A design pattern that provides dependencies to a class from an external container rather than creating them inside the class.

---

**Q2. Which lifetime is used by EF Core DbContext?**

Scoped.

---

**Q3. Can a Singleton depend on a Scoped service?**

No. It can lead to runtime errors because the scoped service may be disposed while the singleton still holds a reference.

---

**Q4. Which DI lifetime is best for repositories and business services?**

Scoped.
