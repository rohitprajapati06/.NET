# Multiple URLs for a Single Resource in ASP.NET Core Web API

ASP.NET Core allows a single action method to be reached through multiple URLs. This is useful for:

* Supporting old API routes for backward compatibility
* Providing aliases for the same resource
* API versioning
* Gradual migration to a new URL structure

---

## 1. Multiple Route Attributes on the Same Action

```csharp
[ApiController]
[Route("api/hospitals")]
public class HospitalsController : ControllerBase
{
    [HttpGet("{id}")]
    [HttpGet("details/{id}")]
    public IActionResult GetHospital(Guid id)
    {
        return Ok($"Hospital: {id}");
    }
}
```

Both requests invoke the same action:

```http
GET /api/hospitals/1

GET /api/hospitals/details/1
```

---

## 2. Multiple Routes on a Controller

```csharp
[ApiController]
[Route("api/hospitals")]
[Route("api/medical-centers")]
public class HospitalsController : ControllerBase
{
    [HttpGet("{id}")]
    public IActionResult Get(Guid id)
    {
        return Ok();
    }
}
```

Accessible URLs:

```http
GET /api/hospitals/1

GET /api/medical-centers/1
```

---

## 3. Absolute Routes

A route starting with `/` or `~/` ignores the controller-level route.

```csharp
[ApiController]
[Route("api/hospitals")]
public class HospitalsController : ControllerBase
{
    [HttpGet("{id}")]
    [HttpGet("/hospital-info/{id}")]
    public IActionResult Get(Guid id)
    {
        return Ok();
    }
}
```

Supported URLs:

```http
GET /api/hospitals/1

GET /hospital-info/1
```

---

## 4. API Versioning Example

```csharp
[ApiController]
public class HospitalsController : ControllerBase
{
    [HttpGet("api/v1/hospitals/{id}")]
    [HttpGet("api/v2/hospitals/{id}")]
    public IActionResult Get(Guid id)
    {
        return Ok();
    }
}
```

Requests:

```http
GET /api/v1/hospitals/1

GET /api/v2/hospitals/1
```

---

## 5. Named Routes

Named routes don't create additional URLs, but they provide alternate ways to generate links.

```csharp
[HttpGet("{id}", Name = "GetHospital")]
public IActionResult Get(Guid id)
{
    return Ok();
}
```

Later:

```csharp
var url = Url.RouteUrl("GetHospital",
                       new { id = hospital.Id });
```

# Example in Your SmartHealthcare Project

Suppose you want to keep an old endpoint while introducing a new one.

```csharp
[ApiController]
[Route("api/hospitals")]
public class HospitalsController : ControllerBase
{
    [HttpGet("{id:guid}")]
    [HttpGet("hospital/{id:guid}")]
    public async Task<IActionResult> Get(Guid id)
    {
        return Ok(await mediator.Send(
            new GetHospitalByIdQuery(id)));
    }
}
```

Both requests work:

```http
GET /api/hospitals/3fa85f64-5717-4562-b3fc-2c963f66afa6

GET /api/hospitals/hospital/3fa85f64-5717-4562-b3fc-2c963f66afa6
```

# When Should You Use Multiple URLs?

✅ Backward compatibility after changing routes

✅ API version migration

✅ Temporary support for legacy clients

❌ Avoid using multiple URLs just for stylistic reasons, because it increases maintenance effort and can make APIs harder to document.

### Recommendation

For your SmartHealthcare API, it's best to expose a **single canonical REST endpoint** such as:

```http
GET    /api/hospitals/{id}
POST   /api/hospitals
PUT    /api/hospitals/{id}
DELETE /api/hospitals/{id}
```

Use multiple URLs only when you have a concrete compatibility or migration requirement.
