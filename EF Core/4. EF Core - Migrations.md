# EF Core Migrations

EF Core Migrations are used to keep your database schema synchronized with your C# model classes.

Instead of manually creating tables in SQL Server, EF Core can:

- Create tables
- Update columns
- Add relationships
- Modify schema changes
- Track database version history

This is one of the most important features in Entity Framework Core.

---

# Why Migrations Are Needed

Suppose you have this model:

```csharp
public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
}
```

After migration, EF Core creates a `Products` table.

Later you add:

```csharp
public decimal Price { get; set; }
```

Now the database also needs a `Price` column.

Instead of manually altering the database:

```sql
ALTER TABLE Products ADD Price decimal(18,2)
```

EF Core migration automatically generates it.

---

# Migration Workflow

Typical workflow:

```text
1. Create Models
2. Add DbSet in DbContext
3. Create Migration
4. Apply Migration
5. Database Updated
```

Flow:

```text
Model Changes
      ↓
Add-Migration
      ↓
Migration File Generated
      ↓
Update-Database
      ↓
Database Schema Updated
```

# Example Project Setup

## Step 1 — Create Model

```csharp
public class Product
{
    public int Id { get; set; }

    public string Name { get; set; }

    public decimal Price { get; set; }
}
```

---

## Step 2 — Create DbContext

```csharp
using Microsoft.EntityFrameworkCore;

public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options)
        : base(options)
    {
    }

    public DbSet<Product> Products { get; set; }
}
```

---

## Step 3 — Configure Connection String

`appsettings.json`

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=.;Database=EFCoreDemoDb;Trusted_Connection=True;TrustServerCertificate=True;"
  }
}
```

---

## Step 4 — Register DbContext

`Program.cs`

```csharp
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("DefaultConnection")));
```

# Create First Migration

## Package Manager Console

```powershell
Add-Migration InitialCreate
```

## .NET CLI

```bash
dotnet ef migrations add InitialCreate
```

# What Happens Internally

EF Core creates a `Migrations` folder.

Example:

```text
Migrations/
 ├── 20260519123000_InitialCreate.cs
 ├── 20260519123000_InitialCreate.Designer.cs
 └── AppDbContextModelSnapshot.cs
```

---

# Migration File Example

Generated file:

```csharp
protected override void Up(MigrationBuilder migrationBuilder)
{
    migrationBuilder.CreateTable(
        name: "Products",
        columns: table => new
        {
            Id = table.Column<int>(nullable: false)
                .Annotation("SqlServer:Identity", "1, 1"),

            Name = table.Column<string>(nullable: false),

            Price = table.Column<decimal>(type: "decimal(18,2)", nullable: false)
        },
        constraints: table =>
        {
            table.PrimaryKey("PK_Products", x => x.Id);
        });
}
```

---

# Understanding `Up()` and `Down()`

## `Up()`

Applied when migration runs.

```text
Creates/updates database schema
```

## `Down()`

Rollback logic.

```text
Undo migration changes
```

Example:

```csharp
protected override void Down(MigrationBuilder migrationBuilder)
{
    migrationBuilder.DropTable(
        name: "Products");
}
```

# Add New Changes

Suppose you modify model:

```csharp
public string Description { get; set; }
```

Create another migration:

```powershell
Add-Migration AddProductDescription
```

Apply:

```powershell
Update-Database
```

EF Core generates:

```sql
ALTER TABLE Products ADD Description nvarchar(max)
```

automatically.

---

# Migration History Table

EF Core creates:

```text
__EFMigrationsHistory
```

This table tracks:

- Applied migrations
- Migration order
- Current database version

---

# Remove Last Migration

If migration NOT applied yet:

```powershell
Remove-Migration
```

CLI:

```bash
dotnet ef migrations remove
```

---

# Rollback Database

Rollback to specific migration:

```powershell
Update-Database InitialCreate
```

Rollback all migrations:

```powershell
Update-Database 0
```

# Important Commands

| Operation           | PMC                  | CLI                             |
| ------------------- | -------------------- | ------------------------------- |
| Add Migration       | `Add-Migration Name` | `dotnet ef migrations add Name` |
| Apply Migration     | `Update-Database`    | `dotnet ef database update`     |
| Remove Migration    | `Remove-Migration`   | `dotnet ef migrations remove`   |
| List Migrations     | `Get-Migration`      | `dotnet ef migrations list`     |
| Generate SQL Script | `Script-Migration`   | `dotnet ef migrations script`   |

# Common Migration Errors

## 1. No DbContext Found

Error:

```text
Unable to create an object of type 'AppDbContext'
```

Fix:

- Ensure DbContext registered correctly
- Ensure constructor exists

---

## 2. Build Failed

Fix:

- Build solution first
- Resolve compile errors

---

## 3. Connection String Issue

Fix:

- Verify SQL Server running
- Verify connection string

# Best Practices

## Use One Migration Per Logical Change

Good:

```text
AddOrderTable
AddPaymentTable
```

Bad:

```text
FinalChanges2
```

---

## Never Edit Migration History Table Manually

Avoid modifying:

```text
__EFMigrationsHistory
```

---

## Keep Migrations in Source Control

Always commit:

- Migration files
- Snapshot files

## Difference Between EnsureCreated and Migration

| EnsureCreated           | Migration                 |
| ----------------------- | ------------------------- |
| Creates DB once         | Handles schema evolution  |
| No versioning           | Supports version tracking |
| Not production-friendly | Production standard       |
