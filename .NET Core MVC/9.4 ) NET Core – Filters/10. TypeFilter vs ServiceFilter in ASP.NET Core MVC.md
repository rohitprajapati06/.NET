# TypeFilter vs ServiceFilter in ASP.NET Core MVC

Both **TypeFilter** and **ServiceFilter** are used to apply filters that depend on **Dependency Injection (DI)** in ASP.NET Core MVC.

They allow filters to receive services such as:

* `ILogger`
* `IConfiguration`
* `DbContext`
* Custom services

The main difference is **how the filter instance is created**.

---

# Why Do We Need Them?

Consider a filter:

```csharp
public class LoggingActionFilter : IActionFilter
{
    private readonly ILogger<LoggingActionFilter> _logger;

    public LoggingActionFilter(
        ILogger<LoggingActionFilter> logger)
    {
        _logger = logger;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _logger.LogInformation("Action Started");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
        _logger.LogInformation("Action Completed");
    }
}
```

This filter requires `ILogger`.

You cannot use:

```csharp
[LoggingActionFilter]
public IActionResult Index()
{
    return View();
}
```

because attributes cannot directly receive services from DI.

ASP.NET Core provides:

* `ServiceFilter`
* `TypeFilter`

to solve this problem.

---

# ServiceFilter

`ServiceFilter` tells ASP.NET Core:

> "Resolve this filter from the Dependency Injection container."

---

## Step 1: Create Filter

```csharp
public class LoggingActionFilter : IActionFilter
{
    private readonly ILogger<LoggingActionFilter> _logger;

    public LoggingActionFilter(
        ILogger<LoggingActionFilter> logger)
    {
        _logger = logger;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _logger.LogInformation("Before Action");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
        _logger.LogInformation("After Action");
    }
}
```

---

## Step 2: Register Filter

### Program.cs

```csharp
builder.Services.AddScoped<LoggingActionFilter>();
```

---

## Step 3: Apply Filter

```csharp
[ServiceFilter(typeof(LoggingActionFilter))]
public IActionResult Index()
{
    return View();
}
```

ASP.NET Core resolves the filter from DI.

---

# TypeFilter

`TypeFilter` tells ASP.NET Core:

> "Create an instance of this filter and resolve its dependencies automatically."

Unlike `ServiceFilter`, explicit registration is not required.

---

## Example

```csharp
[TypeFilter(typeof(LoggingActionFilter))]
public IActionResult Index()
{
    return View();
}
```

No registration needed:

```csharp
// Not Required
builder.Services.AddScoped<LoggingActionFilter>();
```

ASP.NET Core creates the instance automatically.

---

# Constructor Parameters with TypeFilter

A major advantage of `TypeFilter` is passing custom parameters.

---

## Filter

```csharp
public class CustomMessageFilter : IActionFilter
{
    private readonly string _message;

    public CustomMessageFilter(string message)
    {
        _message = message;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        Console.WriteLine(_message);
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
    }
}
```

---

## Apply Filter

```csharp
[TypeFilter(
    typeof(CustomMessageFilter),
    Arguments = new object[]
    {
        "Welcome to TourNest"
    })]
public IActionResult Index()
{
    return View();
}
```

Output:

```text
Welcome to TourNest
```

---

# ServiceFilter Cannot Pass Parameters

This is not possible:

```csharp
[ServiceFilter(typeof(CustomMessageFilter))]
```

because the filter is created directly from DI.

Any constructor parameters must come from DI services.

---

# Internal Working

## ServiceFilter

```text
Request
   ↓
DI Container
   ↓
Get Existing Filter Service
   ↓
Execute Filter
```

---

## TypeFilter

```text
Request
   ↓
Create Filter Instance
   ↓
Resolve Dependencies
   ↓
Pass Custom Arguments
   ↓
Execute Filter
```

# Example with DbContext

## Filter

```csharp
public class AuditFilter : IActionFilter
{
    private readonly ApplicationDbContext _context;

    public AuditFilter(
        ApplicationDbContext context)

        _context = context;

        ActionExecutingContext context)

    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
    }

```

---

## ServiceFilter

```csharp
builder.Services.AddScoped<AuditFilter>();

[ServiceFilter(typeof(AuditFilter))]
public IActionResult Index()
{
    return View();
}
```

---

## TypeFilter

```csharp
[TypeFilter(typeof(AuditFilter))]
public IActionResult Index()

    return View();
```

Both work correctly.

# Comparison Table

| Feature                        | ServiceFilter | TypeFilter    |
| ------------------------------ | ------------- | ------------- |
| Uses DI Container              | Yes           | Yes           |
| Requires Registration          | Yes           | No            |
| Supports Constructor Injection | Yes           | Yes           |
| Supports Custom Parameters     | No            | Yes           |
| Creates Instance Automatically | No            | Yes           |
| Better Performance             | Slightly      | Slightly Less |
| Most Common in Enterprise Apps | Yes           | Sometimes     |

---

# When to Use ServiceFilter

Use `ServiceFilter` when:

✅ Filter is already registered in DI

✅ Filter depends on services

✅ Filter is reused throughout the application

✅ Enterprise applications

Example:

```csharp
builder.Services.AddScoped<AuditFilter>();

[ServiceFilter(typeof(AuditFilter))]
```

---

# When to Use TypeFilter

Use `TypeFilter` when:

✅ No registration desired

✅ Need runtime constructor arguments

✅ One-off filters

Example:

```csharp
[TypeFilter(
    typeof(CustomMessageFilter),
    Arguments = new object[]
    {
        "TourNest"
    })]
```

# Real-World Example

Suppose in your **SmartHealthcare** project:

### Audit Filter

```csharp
public class AuditFilter : IActionFilter
{
    private readonly IAuditService _auditService;

    public AuditFilter(
        IAuditService auditService)
    {
        _auditService = auditService;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _auditService.Log("Action Started");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
    }
}
```

Registration:

```csharp
builder.Services.AddScoped<AuditFilter>();
```

Usage:

```csharp
[ServiceFilter(typeof(AuditFilter))]
public IActionResult CreateDoctor()
{
    return View();
}
```

This is the typical enterprise approach.

# Interview Questions

### What is ServiceFilter?

`ServiceFilter` resolves a filter from the Dependency Injection container.

Example:

```csharp
[ServiceFilter(typeof(MyFilter))]
```

---

### What is TypeFilter?

`TypeFilter` creates a filter instance automatically and resolves dependencies from DI.

Example:

```csharp
[TypeFilter(typeof(MyFilter))]
```

---

### Which one requires DI registration?

**ServiceFilter** requires registration.

```csharp
builder.Services.AddScoped<MyFilter>();
```

---

### Which one supports custom constructor parameters?

**TypeFilter** supports custom parameters through the `Arguments` property.

```csharp
[TypeFilter(
    typeof(MyFilter),
    Arguments = new object[]
    {
        "Custom Value"
    })]
```

---

### Which one is preferred in enterprise applications?

Generally, **ServiceFilter** is preferred because:

* Filters are managed by DI
* Easier to test
* Consistent lifetime management
* Better aligns with Clean Architecture

---

# Quick Summary

| Feature               | ServiceFilter               | TypeFilter                        |
| --------------------- | --------------------------- | --------------------------------- |
| Registration Required | ✅ Yes                       | ❌ No                              |
| Dependency Injection  | ✅ Yes                       | ✅ Yes                             |
| Custom Arguments      | ❌ No                        | ✅ Yes                             |
| Enterprise Preference | ✅ Yes                       | ⚠️ Sometimes                      |
| Best Use Case         | Reusable DI-managed filters | Filters needing runtime arguments |

### Rule of Thumb

```text
Need DI-managed reusable filter?
    → ServiceFilter

Need custom runtime arguments?
    → TypeFilter
```

For your **SmartHealthcare**, **TourNest**, and **La Caffeine** projects, you'll typically use **ServiceFilter** because most filters (logging, auditing, validation, authorization) are registered services and fit naturally into the dependency injection system.
