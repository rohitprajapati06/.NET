# Handling Non-Success HTTP Status Codes in ASP.NET Core MVC

HTTP status codes indicate the result of a request. While **2xx** status codes indicate success, **4xx** and **5xx** status codes represent client-side and server-side errors that should be handled gracefully.

ASP.NET Core MVC provides several ways to handle non-success status codes and display meaningful error pages or responses.

---

# Common HTTP Status Codes

| Status Code | Meaning               |
| ----------- | --------------------- |
| 200         | OK                    |
| 201         | Created               |
| 204         | No Content            |
| 400         | Bad Request           |
| 401         | Unauthorized          |
| 403         | Forbidden             |
| 404         | Not Found             |
| 405         | Method Not Allowed    |
| 409         | Conflict              |
| 500         | Internal Server Error |
| 503         | Service Unavailable   |

---

# Returning Status Codes from Controllers

## 400 Bad Request

```csharp
[HttpPost]
public IActionResult Create(Product product)
{
    if (!ModelState.IsValid)
    {
        return BadRequest(ModelState);
    }

    return Ok();
}
```

Response:

```http
400 Bad Request
```

---

## 401 Unauthorized

```csharp
[HttpGet]
public IActionResult SecureData()
{
    return Unauthorized();
}
```

Response:

```http
401 Unauthorized
```

---

## 403 Forbidden

```csharp
[HttpGet]
public IActionResult AdminOnly()
{
    return Forbid();
}
```

Response:

```http
403 Forbidden
```

---

## 404 Not Found

```csharp
[HttpGet("{id}")]
public IActionResult GetProduct(int id)
{
    var product = _repository.Get(id);

    if (product == null)
    {
        return NotFound();
    }

    return Ok(product);
}
```

Response:

```http
404 Not Found
```

---

## 500 Internal Server Error

```csharp
return StatusCode(500);
```

or

```csharp
return StatusCode(500, "Server Error");
```

# Custom Error Pages

Instead of showing generic browser errors, create custom pages.

## Configure Status Code Pages

### Program.cs

```csharp
app.UseStatusCodePagesWithReExecute("/Error/{0}");
```

`{0}` represents the status code.

Example:

```text
404 → /Error/404
500 → /Error/500
```

---

# Creating Error Controller

```csharp
public class ErrorController : Controller
{
    [Route("Error/{statusCode}")]
    public IActionResult HandleError(int statusCode)
    {
        switch (statusCode)
        {
            case 404:
                return View("NotFound");

            case 403:
                return View("Forbidden");

            case 500:
                return View("ServerError");

            default:
                return View("Error");
        }
    }
}
```

# Global Exception Handling for 500 Errors

Status code pages handle many HTTP errors, but exceptions should be handled separately.

### Program.cs

```csharp
app.UseExceptionHandler("/Error/500");
```

---

### Error Controller

```csharp
[Route("Error/500")]
public IActionResult ServerError()
{
    return View();
}
```

When an unhandled exception occurs:

```csharp
throw new Exception("Database Error");
```

User sees:

```text
500 - Internal Server Error
```

instead of a stack trace.

# Using ProblemDetails (Recommended)

ASP.NET Core provides the `ProblemDetails` standard for API errors.

```csharp
return Problem(
    detail: "Product not found",
    statusCode: 404,
    title: "Not Found");
```

Response:

```json
{
  "type": "about:blank",
  "title": "Not Found",
  "status": 404,
  "detail": "Product not found"
}
```

# Using Custom Middleware for Status Codes

### Middleware

```csharp
public class StatusCodeMiddleware
{
    private readonly RequestDelegate _next;

    public StatusCodeMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task Invoke(HttpContext context)
    {
        await _next(context);

        if (context.Response.StatusCode == 404)
        {
            await context.Response.WriteAsync(
                "Custom 404 Response");
        }
    }
}
```

### Register

```csharp
app.UseMiddleware<StatusCodeMiddleware>();
```

# Best Practice for MVC Applications

### Program.cs

```csharp
app.UseExceptionHandler("/Error/500");

app.UseStatusCodePagesWithReExecute("/Error/{0}");
```

### ErrorController

```csharp
public class ErrorController : Controller
{
    [Route("Error/{statusCode}")]
    public IActionResult HandleError(int statusCode)
    {
        return statusCode switch
        {
            404 => View("NotFound"),
            403 => View("Forbidden"),
            500 => View("ServerError"),
            _ => View("Error")
        };
    }
}
```

This provides a centralized error-handling mechanism.

# Interview Questions

### What is a non-success HTTP status code?

Any status code outside the 2xx range, typically 4xx (client errors) and 5xx (server errors).

---

### How do you create custom error pages in ASP.NET Core MVC?

Use:

```csharp
app.UseStatusCodePagesWithReExecute("/Error/{0}");
```

and create an ErrorController with corresponding views.

---

### What is the difference between `UseExceptionHandler()` and `UseStatusCodePages()`?

| Feature             | UseExceptionHandler | UseStatusCodePages |
| ------------------- | ------------------- | ------------------ |
| Handles Exceptions  | Yes                 | No                 |
| Handles 404/403/401 | No                  | Yes                |
| Handles 500 Errors  | Yes                 | No                 |

---

### What is the recommended API error response format?

Use **ProblemDetails** because it follows RFC 7807 standards and provides a consistent structure for API errors.

## Enterprise ASP.NET Core Setup

For projects like **SmartHealthcare**, **TourNest**, or **La Caffeine**, a common production setup is:

```text
Global Exception Middleware
          +
Serilog Logging
          +
ProblemDetails
          +
UseStatusCodePagesWithReExecute
          +
Custom Error Pages
```

This gives clean error handling for both MVC views and Web API endpoints.
