# Filters in ASP.NET Core MVC

Filters in ASP.NET Core MVC are components that allow you to execute code **before or after specific stages** of the request-processing pipeline. They help implement **cross-cutting concerns** such as logging, authorization, caching, exception handling, and validation without duplicating code across controllers and actions.

## Why Use Filters?

Instead of writing the same code in every action method, filters provide a centralized way to handle common tasks:

* Authentication and Authorization
* Logging
* Exception Handling
* Input Validation
* Response Modification
* Performance Monitoring
* Caching

---

# ASP.NET Core MVC Request Pipeline with Filters

```text
Request
   │
   ▼
Authorization Filter
   │
   ▼
Resource Filter
   │
   ▼
Action Filter (Before)
   │
   ▼
Action Method
   │
   ▼
Action Filter (After)
   │
   ▼
Result Filter (Before)
   │
   ▼
View Result / JSON Result
   │
   ▼
Result Filter (After)
   │
   ▼
Response
```

If an exception occurs:

```text
Exception Filter
        │
        ▼
 Error Handling
```

---

# Types of Filters

ASP.NET Core MVC provides five main filter types:

| Filter Type          | Purpose                                  |
| -------------------- | ---------------------------------------- |
| Authorization Filter | Checks user permissions                  |
| Resource Filter      | Executes before and after model binding  |
| Action Filter        | Executes before and after action methods |
| Result Filter        | Executes before and after action results |
| Exception Filter     | Handles exceptions                       |

---

# 1. Authorization Filters

Authorization filters run first.

They determine whether a user is allowed to access a resource.

### Built-in Example

```csharp
[Authorize]
public class ProductController : Controller
{
    public IActionResult Index()
    {
        return View();
    }
}
```

### Role-Based Authorization

```csharp
[Authorize(Roles = "Admin")]
public IActionResult AdminPanel()
{
    return View();
}
```

Only Admin users can access the action.

---

# 2. Resource Filters

Resource filters execute:

* Before model binding
* After action execution

Useful for:

* Caching
* Performance monitoring
* Short-circuiting requests

### Custom Resource Filter

```csharp
using Microsoft.AspNetCore.Mvc.Filters;

public class CustomResourceFilter : IResourceFilter
{
    public void OnResourceExecuting(ResourceExecutingContext context)
    {
        Console.WriteLine("Before Resource Execution");
    }

    public void OnResourceExecuted(ResourceExecutedContext context)
    {
        Console.WriteLine("After Resource Execution");
    }
}
```

### Apply Filter

```csharp
[ServiceFilter(typeof(CustomResourceFilter))]
public IActionResult Index()
{
    return View();
}
```

---

# 3. Action Filters

Action filters execute:

* Before an action method
* After an action method

Most commonly used filter type.

### Example

```csharp
using Microsoft.AspNetCore.Mvc.Filters;

public class LoggingActionFilter : IActionFilter
{
    public void OnActionExecuting(ActionExecutingContext context)
    {
        Console.WriteLine("Action is starting");
    }

    public void OnActionExecuted(ActionExecutedContext context)
    {
        Console.WriteLine("Action has completed");
    }
}
```

### Apply to Action

```csharp
[ServiceFilter(typeof(LoggingActionFilter))]
public IActionResult Index()
{
    return View();
}
```

### Output

```text
Action is starting
Action Method Executes
Action has completed
```

---

# Async Action Filters

For asynchronous operations:

```csharp
using Microsoft.AspNetCore.Mvc.Filters;

public class AsyncLoggingFilter : IAsyncActionFilter
{
    public async Task OnActionExecutionAsync(
        ActionExecutingContext context,
        ActionExecutionDelegate next)
    {
        Console.WriteLine("Before Action");

        await next();

        Console.WriteLine("After Action");
    }
}
```

---

# 4. Result Filters

Result filters execute before and after the action result.

Useful for:

* Modifying response headers
* Logging responses
* Adding metadata

### Example

```csharp
public class CustomResultFilter : IResultFilter
{
    public void OnResultExecuting(ResultExecutingContext context)
    {
        Console.WriteLine("Before Result");
    }

    public void OnResultExecuted(ResultExecutedContext context)
    {
        Console.WriteLine("After Result");
    }
}
```

### Apply Filter

```csharp
[ServiceFilter(typeof(CustomResultFilter))]
public IActionResult Index()
{
    return View();
}
```

---

# 5. Exception Filters

Exception filters handle unhandled exceptions.

### Example

```csharp
using Microsoft.AspNetCore.Mvc.Filters;

public class CustomExceptionFilter : IExceptionFilter
{
    public void OnException(ExceptionContext context)
    {
        context.Result = new JsonResult(new
        {
            Message = "Something went wrong"
        });

        context.ExceptionHandled = true;
    }
}
```

### Apply

```csharp
[ServiceFilter(typeof(CustomExceptionFilter))]
public IActionResult Index()
{
    throw new Exception("Error");
}
```

### Response

```json
{
  "message": "Something went wrong"
}
```

# Filter Scope

Filters can be applied at three levels.

## Action Level

```csharp
[LoggingActionFilter]
public IActionResult Index()
{
    return View();
}
```

Applies only to the action.

---

## Controller Level

```csharp
[LoggingActionFilter]
public class ProductController : Controller
{
}
```

Applies to all actions within the controller.

---

## Global Level

### Program.cs

```csharp
builder.Services.AddControllersWithViews(options =>
{
    options.Filters.Add<LoggingActionFilter>();
});
```

Applies to every controller and action.

# ServiceFilter vs TypeFilter

## ServiceFilter

Uses Dependency Injection container.

```csharp
[ServiceFilter(typeof(LoggingActionFilter))]
```

Register service:

```csharp
builder.Services.AddScoped<LoggingActionFilter>();
```

---

## TypeFilter

Creates instance automatically.

```csharp
[TypeFilter(typeof(LoggingActionFilter))]
```

No explicit registration required.

# Filter Execution Order

Execution order:

```text
Authorization Filter
       ↓
Resource Filter
       ↓
Action Filter (Before)
       ↓
Action Method
       ↓
Action Filter (After)
       ↓
Result Filter (Before)
       ↓
Result Execution
       ↓
Result Filter (After)
```

If an exception occurs:

```text
Exception Filter
       ↓
Error Response
```

# Real-World Examples

## Logging Requests

```csharp
public class RequestLoggingFilter : IActionFilter
{
    private readonly ILogger<RequestLoggingFilter> _logger;

    public RequestLoggingFilter(
        ILogger<RequestLoggingFilter> logger)
    {
        _logger = logger;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _logger.LogInformation(
            $"Request Started: {context.HttpContext.Request.Path}");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
        _logger.LogInformation(
            $"Request Completed: {context.HttpContext.Request.Path}");
    }
}
```

---

## Adding Response Header

```csharp
public class CustomHeaderFilter : ResultFilterAttribute
{
    public override void OnResultExecuting(
        ResultExecutingContext context)
    {
        context.HttpContext.Response.Headers.Add(
            "Application",
            "TourNest");
    }
}
```

# Filters vs Middleware

| Feature                | Filters                            | Middleware                                         |
| ---------------------- | ---------------------------------- | -------------------------------------------------- |
| Scope                  | MVC Only                           | Entire Application                                 |
| Access Action Context  | Yes                                | No                                                 |
| Runs Before Controller | Yes                                | Yes                                                |
| Runs After Controller  | Yes                                | Yes                                                |
| Best For               | Validation, Authorization, Logging | Authentication, Routing, Global Exception Handling |

### Use Middleware When

* Authentication
* Request Logging
* Global Exception Handling
* Response Compression

### Use Filters When

* Action Validation
* Authorization
* Result Manipulation
* MVC-specific Logic

# Interview Questions

### What are Filters in ASP.NET Core MVC?

Filters are components that execute code before or after specific stages of the MVC request pipeline.

### How many types of filters are available?

1. Authorization Filter
2. Resource Filter
3. Action Filter
4. Result Filter
5. Exception Filter

### Difference between Middleware and Filters?

Middleware works for the entire application pipeline, while filters work only within the MVC pipeline.

### Difference between ServiceFilter and TypeFilter?

* **ServiceFilter** uses Dependency Injection and requires registration.
* **TypeFilter** creates instances automatically and doesn't require registration.

### Which filter is most commonly used?

**Action Filters** are the most commonly used because they allow logic execution before and after controller actions.
