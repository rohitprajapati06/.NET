# TypeFilter vs ServiceFilter in ASP.NET Core MVC

Both **TypeFilter** and **ServiceFilter** are used when you want to apply filters that require **Dependency Injection (DI)** in ASP.NET Core MVC.

For example, if your filter needs:

* `ILogger`
* `ApplicationDbContext`
* `IEmailService`
* `IUserService`

you cannot directly use the filter as a normal attribute because attributes do not support constructor injection.

ASP.NET Core provides:

* **ServiceFilter**
* **TypeFilter**

to solve this problem.

---

# Quick Difference

| Feature                        | ServiceFilter | TypeFilter           |
| ------------------------------ | ------------- | -------------------- |
| Requires DI Registration       | ✅ Yes         | ❌ No                 |
| Uses Dependency Injection      | ✅ Yes         | ✅ Yes                |
| Supports Constructor Arguments | ❌ No          | ✅ Yes                |
| Creates Instance Automatically | ❌ No          | ✅ Yes                |
| Enterprise Preference          | ✅ Yes         | ⚠️ For special cases |

---

# Why Normal Filter Attributes Fail

Suppose you have:

```csharp
public class LoggingFilter : IActionFilter
{
    private readonly ILogger<LoggingFilter> _logger;

    public LoggingFilter(
        ILogger<LoggingFilter> logger)
    {
        _logger = logger;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _logger.LogInformation("Action Started");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
        _logger.LogInformation("Action Finished");
    }
}
```

This won't work:

```csharp
[LoggingFilter]
public IActionResult Index()
{
    return View();
}
```

because attributes cannot receive services from the DI container.

---

# ServiceFilter

`ServiceFilter` tells ASP.NET Core:

> "Resolve this filter from the Dependency Injection container."

---

## Step 1: Create Filter

```csharp
public class LoggingFilter : IActionFilter
{
    private readonly ILogger<LoggingFilter> _logger;

    public LoggingFilter(
        ILogger<LoggingFilter> logger)
    {
        _logger = logger;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _logger.LogInformation("Before Action");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
        _logger.LogInformation("After Action");
    }
}
```

---

## Step 2: Register Filter

### Program.cs

```csharp
builder.Services.AddScoped<LoggingFilter>();
```

---

## Step 3: Apply Filter

```csharp
[ServiceFilter(typeof(LoggingFilter))]
public IActionResult Index()
{
    return View();
}
```

ASP.NET Core retrieves the filter from DI and executes it.

---

# ServiceFilter Lifecycle

```text
Request
   ↓
DI Container
   ↓
Resolve LoggingFilter
   ↓
Execute Filter
```

The filter's lifetime follows its registration:

```csharp
builder.Services.AddScoped<LoggingFilter>();
```

or

```csharp
builder.Services.AddTransient<LoggingFilter>();
```

---

# TypeFilter

`TypeFilter` tells ASP.NET Core:

> "Create the filter instance and resolve any dependencies automatically."

No registration is required.

---

## Example

```csharp
[TypeFilter(typeof(LoggingFilter))]
public IActionResult Index()
{
    return View();
}
```

No need for:

```csharp
builder.Services.AddScoped<LoggingFilter>();
```

ASP.NET Core creates the filter instance internally.

---

# TypeFilter Lifecycle

```text
Request
   ↓
Create Filter Instance
   ↓
Resolve Dependencies
   ↓
Execute Filter
```

---

# Passing Constructor Parameters

This is where **TypeFilter** becomes very useful.

---

## Filter

```csharp
public class MessageFilter : IActionFilter
{
    private readonly string _message;

    public MessageFilter(string message)
    {
        _message = message;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        Console.WriteLine(_message);
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
    }
}
```

---

## Apply Using TypeFilter

```csharp
[TypeFilter(
    typeof(MessageFilter),
    Arguments = new object[]
    {
        "Welcome to TourNest"
    })]
public IActionResult Index()
{
    return View();
}
```

Output:

```text
Welcome to TourNest
```

---

# Why ServiceFilter Cannot Do This

`ServiceFilter` resolves the filter from DI:

```csharp
[ServiceFilter(typeof(MessageFilter))]
```

The DI container cannot supply arbitrary runtime values like:

```text
"Welcome to TourNest"
```

unless they are registered services.

Therefore:

```text
Custom Constructor Arguments
            ↓
Use TypeFilter
```

---

# Example with DbContext

### Filter

```csharp
public class AuditFilter : IActionFilter
{
    private readonly ApplicationDbContext _db;

    public AuditFilter(
        ApplicationDbContext db)
    {
        _db = db;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
    }
}
```

---

## ServiceFilter Version

```csharp
builder.Services.AddScoped<AuditFilter>();

[ServiceFilter(typeof(AuditFilter))]
public IActionResult Create()
{
    return View();
}
```

---

## TypeFilter Version

```csharp
[TypeFilter(typeof(AuditFilter))]
public IActionResult Create()
{
    return View();
}
```

Both work because `ApplicationDbContext` comes from DI.

---

# ServiceFilter vs TypeFilter Internally

### ServiceFilter

```text
Attribute
    ↓
Ask DI Container
    ↓
Get Registered Filter
    ↓
Execute
```

### TypeFilter

```text
Attribute
    ↓
Create Object
    ↓
Resolve DI Dependencies
    ↓
Inject Custom Arguments
    ↓
Execute
```

---

# When to Use ServiceFilter

Use ServiceFilter when:

✅ Filter is reused throughout the application

✅ Filter is registered in DI

✅ Filter only depends on services

✅ You follow Clean Architecture

Example:

```csharp
builder.Services.AddScoped<AuditFilter>();

[ServiceFilter(typeof(AuditFilter))]
```

---

# When to Use TypeFilter

Use TypeFilter when:

✅ Runtime values are needed

✅ Filter is action-specific

✅ You don't want DI registration

Example:

```csharp
[TypeFilter(
    typeof(MessageFilter),
    Arguments = new object[]
    {
        "Special Promotion"
    })]
```

# Real-World Example (SmartHealthcare)

Suppose you create an audit filter:

```csharp
public class AuditFilter : IActionFilter
{
    private readonly IAuditService _auditService;

    public AuditFilter(
        IAuditService auditService)
    {
        _auditService = auditService;
    }

    public void OnActionExecuting(
        ActionExecutingContext context)
    {
        _auditService.Log("Action Started");
    }

    public void OnActionExecuted(
        ActionExecutedContext context)
    {
    }
}
```

Register:

```csharp
builder.Services.AddScoped<AuditFilter>();
```

Usage:

```csharp
[ServiceFilter(typeof(AuditFilter))]
public IActionResult ApproveDoctor()
{
    return View();
}
```

This is the most common enterprise pattern.

# Interview Questions

### What is ServiceFilter?

`ServiceFilter` resolves a filter from the Dependency Injection container.

```csharp
[ServiceFilter(typeof(MyFilter))]
```

---

### What is TypeFilter?

`TypeFilter` creates a filter instance and resolves dependencies automatically.

```csharp
[TypeFilter(typeof(MyFilter))]
```

---

### Which one requires DI registration?

**ServiceFilter**

```csharp
builder.Services.AddScoped<MyFilter>();
```

---

### Which one supports constructor arguments?

**TypeFilter**

```csharp
[TypeFilter(
    typeof(MyFilter),
    Arguments = new object[]
    {
        "Custom Value"
    })]
```

---

### Which one is preferred in enterprise applications?

Usually **ServiceFilter**, because:

* Lifetime is managed by DI
* Easier to test
* Consistent dependency management
* Fits Clean Architecture and CQRS patterns

# Recommended for Your Projects

For your **SmartHealthcare**, **TourNest**, and **La Caffeine** projects:

```text
Logging Filter        → ServiceFilter
Audit Filter          → ServiceFilter
Validation Filter     → ServiceFilter
Authorization Filter  → ServiceFilter
```

Use **TypeFilter** only when you need to pass custom values at runtime.

### Simple Rule

```text
Filter uses registered services only?
    → ServiceFilter

Filter needs custom constructor arguments?
    → TypeFilter
```
