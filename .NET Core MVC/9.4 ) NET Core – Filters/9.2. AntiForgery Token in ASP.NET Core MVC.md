# AntiForgery Token in ASP.NET Core MVC

An **AntiForgery Token** is a security mechanism used to protect ASP.NET Core MVC applications against **Cross-Site Request Forgery (CSRF)** attacks.

It ensures that a form submission originates from your application and not from a malicious website.

---

# What is CSRF?

**Cross-Site Request Forgery (CSRF)** is an attack where a malicious website tricks an authenticated user into performing unwanted actions on another website.

### Example

Suppose a user is logged into your banking application.

Your application:

```text
https://mybank.com
```

Attacker's website:

```text
https://hacker.com
```

The attacker creates a hidden form:

```html
<form action="https://mybank.com/TransferMoney" method="post">
    <input type="hidden" name="amount" value="10000" />
    <input type="hidden" name="account" value="123456" />
</form>

<script>
    document.forms[0].submit();
</script>
```

If the victim is already logged in to `mybank.com`, the browser automatically sends authentication cookies.

Result:

```text
Money Transfer Executed
```

without the user's knowledge.

---

# How AntiForgery Token Prevents CSRF

ASP.NET Core generates:

1. A hidden token in the form
2. A corresponding token in a cookie

When the form is submitted:

```text
Request Token
      +
Cookie Token
      ↓
Validation
```

If both tokens match:

```text
Request Accepted
```

Otherwise:

```text
400 Bad Request
```

# Using AntiForgery Token

## Step 1: Create a Form

```html
<form asp-action="Create" method="post">

    <input type="text" name="Name" />

    <button type="submit">
        Save
    </button>

</form>
```

---

## Step 2: Add AntiForgery Token

```html
<form asp-action="Create" method="post">

    @Html.AntiForgeryToken()

    <input type="text" name="Name" />

    <button type="submit">
        Save
    </button>

</form>
```

Generated HTML:

```html
<input
    name="__RequestVerificationToken"
    type="hidden"
    value="random-token" />
```

---

# Validate Token in Controller

Use:

```csharp
[HttpPost]
[ValidateAntiForgeryToken]
public IActionResult Create(Product product)
{
    return View();
}
```

ASP.NET Core validates the token automatically.

# What Happens During Validation?

### Request

```text
Cookie:
Token123

Hidden Field:
Token123
```

Validation:

```text
Token123 == Token123
```

Result:

```text
Request Allowed
```

---

### Invalid Request

```text
Cookie:
Token123

Hidden Field:
Token999
```

Validation:

```text
Token123 != Token999
```

Result:

```http
400 Bad Request
```

# AutoValidateAntiforgeryToken

Instead of decorating every action:

```csharp
[ValidateAntiForgeryToken]
```

Use globally.

### Program.cs

```csharp
builder.Services.AddControllersWithViews(options =>
{
    options.Filters.Add(
        new AutoValidateAntiforgeryTokenAttribute());
});
```

Now every unsafe HTTP method is validated automatically.

---

# Difference Between ValidateAntiForgeryToken and AutoValidateAntiforgeryToken

| Attribute                    | Applies To                 |
| ---------------------------- | -------------------------- |
| ValidateAntiForgeryToken     | Specific action/controller |
| AutoValidateAntiforgeryToken | Entire application         |

---

## ValidateAntiForgeryToken

```csharp
[HttpPost]
[ValidateAntiForgeryToken]
public IActionResult Create(Product product)
{
    return View();
}
```

Only this action validates the token.

---

## AutoValidateAntiforgeryToken

```csharp
builder.Services.AddControllersWithViews(options =>
{
    options.Filters.Add(
        new AutoValidateAntiforgeryTokenAttribute());
});
```

All unsafe requests are validated automatically.

---

# Ignore AntiForgery Validation

Use:

```csharp
[IgnoreAntiforgeryToken]
```

Example:

```csharp
[HttpPost]
[IgnoreAntiforgeryToken]
public IActionResult Webhook()
{
    return Ok();
}
```

Useful for:

* Third-party webhooks
* External callbacks
* Payment gateway callbacks

# AntiForgery with Razor Forms

When using Razor Form Tag Helpers:

```html
<form asp-action="Create" method="post">
</form>
```

ASP.NET Core automatically injects:

```html
<input
    name="__RequestVerificationToken"
    type="hidden" />
```

So manually writing:

```html
@Html.AntiForgeryToken()
```

is often unnecessary.

---

# AntiForgery with AJAX Requests

For AJAX calls, send the token manually.

### Razor View

```html
<input type="hidden"
       id="token"
       value="@Antiforgery.GetAndStoreTokens(Context).RequestToken" />
```

---

### JavaScript

```javascript
fetch("/Product/Create", {
    method: "POST",
    headers: {
        "RequestVerificationToken":
            document.getElementById("token").value
    }
});
```

---

# Configure AntiForgery Options

### Program.cs

```csharp
builder.Services.AddAntiforgery(options =>
{
    options.HeaderName =
        "RequestVerificationToken";
});
```

Custom header name:

```http
RequestVerificationToken: token-value
```

---

# AntiForgery with APIs

Generally:

```text
MVC + Cookies
        ↓
Use AntiForgery Token
```

```text
Web API + JWT
        ↓
AntiForgery Usually Not Needed
```

Reason:

JWT tokens are sent in headers and are not automatically attached by browsers like cookies.

# Common Errors

## Missing Token

```text
The required antiforgery cookie was not provided.
```

Cause:

```html
<form method="post">
```

without token generation.

---

## Invalid Token

```text
The antiforgery token could not be decrypted.
```

Cause:

* Expired token
* Different machine key
* Cookie mismatch

# Interview Questions

### What is an AntiForgery Token?

An AntiForgery Token is a security token used to protect ASP.NET Core MVC applications from CSRF (Cross-Site Request Forgery) attacks.

---

### Which attribute validates AntiForgery Tokens?

```csharp
[ValidateAntiForgeryToken]
```

---

### What does `@Html.AntiForgeryToken()` do?

It generates a hidden form field containing a unique request verification token.

---

### What is the difference between `ValidateAntiForgeryToken` and `AutoValidateAntiforgeryToken`?

| ValidateAntiForgeryToken | AutoValidateAntiforgeryToken |
| ------------------------ | ---------------------------- |
| Applied manually         | Applied globally             |
| Specific actions         | Entire application           |

---

### Is AntiForgery needed for JWT-based APIs?

Generally **No**. AntiForgery protection is primarily needed when using **cookie-based authentication**, because browsers automatically send cookies with requests.
