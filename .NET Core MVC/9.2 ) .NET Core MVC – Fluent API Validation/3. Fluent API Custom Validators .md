# Fluent API Custom Validators in ASP.NET Core MVC

**Custom Validators** in **FluentValidation** allow you to create your own validation rules when built-in validators such as `NotEmpty()`, `EmailAddress()`, or `Length()` are not sufficient.

They are useful for validating:

* Business rules
* Password complexity
* Age restrictions
* Custom formats
* Cross-property validation
* Database-driven validation
* Domain-specific rules

---

# Why Use Custom Validators?

Suppose a password must:

* Be at least 8 characters
* Contain one uppercase letter
* Contain one lowercase letter
* Contain one digit
* Contain one special character

Instead of putting all this logic in a controller, create a custom validator.

---

# Method 1: Using `Must()`

The simplest way to create a custom validator.

## DTO

```csharp
public class RegisterDto
{
    public string Password { get; set; }
}
```

## Validator

```csharp
using FluentValidation;

public class RegisterDtoValidator
    : AbstractValidator<RegisterDto>
{
    public RegisterDtoValidator()
    {
        RuleFor(x => x.Password)
            .Must(BeStrongPassword)
            .WithMessage(
                "Password must contain uppercase, lowercase, number and special character.");
    }

    private bool BeStrongPassword(string password)
    {
        if (string.IsNullOrWhiteSpace(password))
            return false;

        return password.Any(char.IsUpper)
               && password.Any(char.IsLower)
               && password.Any(char.IsDigit)
               && password.Any(ch => !char.IsLetterOrDigit(ch));
    }
}
```

# Method 4: Using `Custom()`

Provides more control and allows adding multiple validation failures.

```csharp
RuleFor(x => x.Password)
    .Custom((password, context) =>
    {
        if (password.Length < 8)
        {
            context.AddFailure(
                "Password",
                "Password must be at least 8 characters.");
        }

        if (!password.Any(char.IsUpper))
        {
            context.AddFailure(
                "Password",
                "Password must contain an uppercase letter.");
        }
    });
```

### Result

```json
{
  "errors": {
    "Password": [
      "Password must be at least 8 characters.",
      "Password must contain an uppercase letter."
    ]
  }
}
```

# Method 5: Reusable Extension Method

For enterprise applications, create reusable validators.

## Extension Class

```csharp
using FluentValidation;

public static class CustomValidators
{
    public static IRuleBuilderOptions<T, string>
        StrongPassword<T>(
            this IRuleBuilder<T, string> ruleBuilder)
    {
        return ruleBuilder.Must(password =>
        {
            if (string.IsNullOrEmpty(password))
                return false;

            return password.Any(char.IsUpper)
                && password.Any(char.IsLower)
                && password.Any(char.IsDigit)
                && password.Any(ch => !char.IsLetterOrDigit(ch));
        })
        .WithMessage(
            "Password does not meet complexity requirements.");
    }
}
```

## Usage

```csharp
RuleFor(x => x.Password)
    .StrongPassword();
```

# Method 6: Custom Property Validator Class

For highly reusable and complex validation.

## Create Validator

```csharp
using FluentValidation.Validators;

public class StrongPasswordValidator<T>
    : PropertyValidator<T, string>
{
    public override string Name
        => "StrongPasswordValidator";

    public override bool IsValid(
        ValidationContext<T> context,
        string value)
    {
        if (string.IsNullOrWhiteSpace(value))
            return false;

        return value.Any(char.IsUpper)
            && value.Any(char.IsLower)
            && value.Any(char.IsDigit);
    }

    protected override string GetDefaultMessageTemplate(
        string errorCode)
    {
        return "{PropertyName} is not strong enough.";
    }
}
```

---

## Extension Method

```csharp
public static class ValidatorExtensions
{
    public static IRuleBuilderOptions<T, string>
        StrongPassword<T>(
            this IRuleBuilder<T, string> ruleBuilder)
    {
        return ruleBuilder.SetValidator(
            new StrongPasswordValidator<T>());
    }
}
```

---

## Usage

```csharp
RuleFor(x => x.Password)
    .StrongPassword();
```

# Custom Validator with Database Check

Example: Product name must be unique.

```csharp
RuleFor(x => x.Name)
    .Must((dto, productName) =>
    {
        return !_context.Products
            .Any(x => x.Name == productName);
    })
    .WithMessage("Product name already exists.");
```

For database access, prefer async validation:

```csharp
RuleFor(x => x.Name)
    .MustAsync(async (name, cancellation) =>
    {
        return !await _context.Products
            .AnyAsync(
                x => x.Name == name,
                cancellation);
    })
    .WithMessage("Product name already exists.");
```

# Interview Questions

### What is a custom validator in FluentValidation?

A custom validator is a user-defined validation rule used when built-in validators cannot satisfy business requirements.

### What methods are commonly used to create custom validators?

* `Must()`
* `MustAsync()`
* `Custom()`
* `CustomAsync()`
* `SetValidator()`

### When should `MustAsync()` be used?

When validation requires database access, API calls, or other asynchronous operations.

### What is the advantage of extension-method-based validators?

They promote reusability, maintainability, and cleaner validation code across the application.

### What is the difference between `Must()` and `Custom()`?

| Must()                   | Custom()                              |
| ------------------------ | ------------------------------------- |
| Returns true/false       | Can add multiple validation failures  |
| Simpler                  | More flexible                         |
| Single validation result | Multiple validation messages possible |

In enterprise ASP.NET Core applications using **Clean Architecture + CQRS + MediatR**, custom validators are typically placed in the **Application Layer** and executed automatically through a **Validation Pipeline Behavior** before request handlers run.
