# Fluent API in ASP.NET Core MVC

## What is Fluent API?

**Fluent API** is a way to configure entity models in **Entity Framework Core** using C# code instead of Data Annotations.

It is used inside the `OnModelCreating()` method of the `DbContext` class.

### Why use Fluent API?

Data Annotations have limitations. Fluent API provides:

* More configuration options
* Better separation of concerns
* Complex relationship mapping
* Database-specific configurations
* Composite keys support
* Precise control over table and column mappings

---

# Data Annotations vs Fluent API

### Data Annotations

```csharp
public class Employee
{
    public int Id { get; set; }

    [Required]
    [MaxLength(100)]
    public string Name { get; set; }
}
```

### Fluent API

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Employee>()
        .Property(e => e.Name)
        .IsRequired()
        .HasMaxLength(100);
}
```

---

# Where is Fluent API Written?

Inside your DbContext class:

```csharp
public class ApplicationDbContext : DbContext
{
    public ApplicationDbContext(
        DbContextOptions<ApplicationDbContext> options)
        : base(options)
    {
    }

    public DbSet<Employee> Employees { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        base.OnModelCreating(modelBuilder);

        // Fluent API configurations
    }
}
```

# Common Fluent API Configurations

---

## 1. Configure Table Name

### Entity

```csharp
public class Employee
{
    public int Id { get; set; }
    public string Name { get; set; }
}
```

### Fluent API

```csharp
modelBuilder.Entity<Employee>()
    .ToTable("tblEmployees");
```

### Database Table

```sql
tblEmployees
```

---

## 2. Configure Primary Key

### Single Key

```csharp
modelBuilder.Entity<Employee>()
    .HasKey(e => e.Id);
```

---

## 3. Configure Composite Key

Sometimes multiple columns together form the primary key.

```csharp
public class StudentCourse
{
    public int StudentId { get; set; }
    public int CourseId { get; set; }
}
```

```csharp
modelBuilder.Entity<StudentCourse>()
    .HasKey(sc => new
    {
        sc.StudentId,
        sc.CourseId
    });
```

Generated SQL:

```sql
PRIMARY KEY(StudentId, CourseId)
```

---

## 4. Configure Required Property

```csharp
modelBuilder.Entity<Employee>()
    .Property(e => e.Name)
    .IsRequired();
```

Equivalent to:

```csharp
[Required]
public string Name { get; set; }
```

---

## 5. Configure Maximum Length

```csharp
modelBuilder.Entity<Employee>()
    .Property(e => e.Name)
    .HasMaxLength(100);
```

Equivalent to:

```csharp
[StringLength(100)]
public string Name { get; set; }
```

---

## 6. Configure Column Name

```csharp
modelBuilder.Entity<Employee>()
    .Property(e => e.Name)
    .HasColumnName("EmployeeName");
```

Generated SQL:

```sql
EmployeeName
```

---

## 7. Configure Column Type

```csharp
modelBuilder.Entity<Employee>()
    .Property(e => e.Salary)
    .HasColumnType("decimal(18,2)");
```

Generated SQL:

```sql
Salary decimal(18,2)
```

---

## 8. Configure Default Value

```csharp
modelBuilder.Entity<Employee>()
    .Property(e => e.IsActive)
    .HasDefaultValue(true);
```

SQL:

```sql
DEFAULT 1
```

---

## 9. Configure Unique Constraint

```csharp
modelBuilder.Entity<Employee>()
    .HasIndex(e => e.Email)
    .IsUnique();
```

Generated SQL:

```sql
CREATE UNIQUE INDEX
```

# One-to-One Relationship

## Models

```csharp
public class User
{
    public int Id { get; set; }

    public UserProfile Profile { get; set; }
}

public class UserProfile
{
    public int Id { get; set; }

    public int UserId { get; set; }

    public User User { get; set; }
}
```

### Fluent API

```csharp
modelBuilder.Entity<User>()
    .HasOne(u => u.Profile)
    .WithOne(p => p.User)
    .HasForeignKey<UserProfile>(p => p.UserId);
```

---

# One-to-Many Relationship

Most common relationship.

## Models

```csharp
public class Department
{
    public int Id { get; set; }

    public ICollection<Employee> Employees { get; set; }
}

public class Employee
{
    public int Id { get; set; }

    public int DepartmentId { get; set; }

    public Department Department { get; set; }
}
```

### Fluent API

```csharp
modelBuilder.Entity<Employee>()
    .HasOne(e => e.Department)
    .WithMany(d => d.Employees)
    .HasForeignKey(e => e.DepartmentId);
```

---

# Many-to-Many Relationship

### Models

```csharp
public class Student
{
    public int Id { get; set; }

    public ICollection<Course> Courses { get; set; }
}

public class Course
{
    public int Id { get; set; }

    public ICollection<Student> Students { get; set; }
}
```

### Fluent API

```csharp
modelBuilder.Entity<Student>()
    .HasMany(s => s.Courses)
    .WithMany(c => c.Students)
    .UsingEntity(j => j.ToTable("StudentCourses"));
```

Generated Table:

```sql
StudentCourses
(
    StudentId,
    CourseId
)
```

---

# Configure Cascade Delete

When parent is deleted, child records are also deleted.

```csharp
modelBuilder.Entity<Employee>()
    .HasOne(e => e.Department)
    .WithMany(d => d.Employees)
    .HasForeignKey(e => e.DepartmentId)
    .OnDelete(DeleteBehavior.Cascade);
```

Options:

```csharp
DeleteBehavior.Cascade
DeleteBehavior.Restrict
DeleteBehavior.NoAction
DeleteBehavior.SetNull
```

---

# Ignore a Property

If you don't want a property stored in the database:

```csharp
public class Employee
{
    public int Id { get; set; }

    public string FullName { get; set; }
}
```

```csharp
modelBuilder.Entity<Employee>()
    .Ignore(e => e.FullName);
```

No column will be created.

# Configure Seed Data

```csharp
modelBuilder.Entity<Role>().HasData(
    new Role
    {
        Id = 1,
        Name = "Admin"
    },
    new Role
    {
        Id = 2,
        Name = "User"
    });
```

After migration:

```sql
INSERT INTO Roles
```

automatically executes.

# Using Separate Configuration Classes

For large projects, create separate configuration files.

## EmployeeConfiguration.cs

```csharp
using Microsoft.EntityFrameworkCore;
using Microsoft.EntityFrameworkCore.Metadata.Builders;

public class EmployeeConfiguration :
    IEntityTypeConfiguration<Employee>
{
    public void Configure(
        EntityTypeBuilder<Employee> builder)
    {
        builder.ToTable("Employees");

        builder.HasKey(e => e.Id);

        builder.Property(e => e.Name)
               .IsRequired()
               .HasMaxLength(100);

        builder.HasIndex(e => e.Email)
               .IsUnique();
    }
}
```

## Register Configuration

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.ApplyConfigurationsFromAssembly(
        typeof(ApplicationDbContext).Assembly);
}
```

This is the preferred approach in **Clean Architecture**, **CQRS**, and enterprise-level ASP.NET Core projects.

# Interview Questions

### What is Fluent API?

A method of configuring Entity Framework Core entities using code inside `OnModelCreating()` instead of Data Annotations.

---

### When should Fluent API be preferred over Data Annotations?

When:

* Complex relationships are required
* Composite keys are needed
* Advanced database configuration is required
* Large enterprise projects need centralized configuration

---

### Can Fluent API override Data Annotations?

Yes.

If both are present, **Fluent API takes precedence**.

---

### What method is used for Fluent API configuration?

```csharp
OnModelCreating(ModelBuilder modelBuilder)
```

---

### What interface is used for separate entity configurations?

```csharp
IEntityTypeConfiguration<T>
```

---

# Best Practice

For enterprise ASP.NET Core applications (like your healthcare and booking systems):

* Keep entities clean.
* Avoid excessive Data Annotations.
* Use `IEntityTypeConfiguration<T>` for each entity.
* Register configurations using:

```csharp
modelBuilder.ApplyConfigurationsFromAssembly(
    typeof(ApplicationDbContext).Assembly);
```

This keeps the Domain layer clean and makes Entity Framework configurations easier to maintain and scale.
