# Data Annotations vs Fluent API in ASP.NET Core MVC

Both **Data Annotations** and **Fluent API** are used to configure models and validation in ASP.NET Core applications. However, they serve different purposes and are suitable for different scenarios.

---

# What are Data Annotations?

Data Annotations are attributes applied directly to model properties or classes.

### Example

```csharp
using System.ComponentModel.DataAnnotations;

public class Employee
{
    public int Id { get; set; }

    [Required]
    [StringLength(100)]
    [EmailAddress]
    public string Email { get; set; }
}
```

The attributes define validation and some database configuration.

---

# What is Fluent API?

Fluent API is a code-based configuration approach provided by Entity Framework Core through the `ModelBuilder` class.

### Example

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Employee>()
        .Property(e => e.Email)
        .IsRequired()
        .HasMaxLength(100);
}
```

---

# Key Differences

| Feature                        | Data Annotations      | Fluent API                                             |
| ------------------------------ | --------------------- | ------------------------------------------------------ |
| Configuration Location         | Inside model class    | Inside DbContext or Configuration Classes              |
| Readability                    | Easy for simple rules | Better for complex configurations                      |
| Learning Curve                 | Easy                  | Moderate                                               |
| Supports Validation            | Yes                   | No (EF Core Fluent API is for mapping, not validation) |
| Supports Complex Relationships | Limited               | Yes                                                    |
| Composite Keys                 | No                    | Yes                                                    |
| Table Mapping                  | Limited               | Full support                                           |
| Column Mapping                 | Limited               | Full support                                           |
| Separation of Concerns         | Mixed with model      | Clean separation                                       |
| Enterprise Projects            | Less preferred        | Preferred                                              |

---

# Validation Example

## Data Annotations

```csharp
public class RegisterDto
{
    [Required]
    [EmailAddress]
    public string Email { get; set; }

    [Required]
    [MinLength(8)]
    public string Password { get; set; }
}
```

ASP.NET Core automatically validates the model.

---

## FluentValidation (Often Called Fluent Validation API)

```csharp
public class RegisterDtoValidator
    : AbstractValidator<RegisterDto>
{
    public RegisterDtoValidator()
    {
        RuleFor(x => x.Email)
            .NotEmpty()
            .EmailAddress();

        RuleFor(x => x.Password)
            .MinimumLength(8);
    }
}
```

This is different from EF Core Fluent API and is commonly used in Clean Architecture projects.

---

# Database Mapping Example

## Data Annotations

```csharp
public class Product
{
    [Key]
    public int Id { get; set; }

    [Column("ProductName")]
    [MaxLength(100)]
    public string Name { get; set; }
}
```

---

## Fluent API

```csharp
modelBuilder.Entity<Product>()
    .Property(p => p.Name)
    .HasColumnName("ProductName")
    .HasMaxLength(100);
```

---

# Relationship Configuration

## Data Annotations

```csharp
public class Employee
{
    public int Id { get; set; }

    [ForeignKey("Department")]
    public int DepartmentId { get; set; }

    public Department Department { get; set; }
}
```

Works for simple relationships.

---

## Fluent API

```csharp
modelBuilder.Entity<Employee>()
    .HasOne(e => e.Department)
    .WithMany(d => d.Employees)
    .HasForeignKey(e => e.DepartmentId);
```

Better for complex relationship mapping.

---

# Composite Key Example

## Data Annotations

❌ Not Supported

```csharp
public class StudentCourse
{
    public int StudentId { get; set; }
    public int CourseId { get; set; }
}
```

You cannot create a composite key using Data Annotations.

---

## Fluent API

✅ Supported

```csharp
modelBuilder.Entity<StudentCourse>()
    .HasKey(sc => new
    {
        sc.StudentId,
        sc.CourseId
    });
```

---

# Unique Constraint Example

## Data Annotations

❌ No direct support

---

## Fluent API

✅ Supported

```csharp
modelBuilder.Entity<User>()
    .HasIndex(x => x.Email)
    .IsUnique();
```

---

# Cascade Delete Example

## Data Annotations

❌ Not Supported

---

## Fluent API

✅ Supported

```csharp
modelBuilder.Entity<Order>()
    .HasOne(o => o.Customer)
    .WithMany(c => c.Orders)
    .OnDelete(DeleteBehavior.Cascade);
```

# Enterprise Project Structure

## Data Annotations

```text
Domain
 └── Entities
      └── User.cs
```

Validation and mapping attributes are mixed with business entities.

---

## Fluent API

```text
Domain
 └── Entities
      └── User.cs

Infrastructure
 └── Persistence
      └── Configurations
           └── UserConfiguration.cs
```

Entities remain clean.

# Which One Should You Use?

## Use Data Annotations When

* Small projects
* Learning ASP.NET Core
* Simple CRUD applications
* Few validation rules
* Rapid development

Example:

```csharp
[Required]
[StringLength(50)]
public string Name { get; set; }
```

---

## Use Fluent API When

* Clean Architecture
* CQRS + MediatR
* Large applications
* Complex relationships
* Composite keys
* Advanced database configurations

Example:

```csharp
builder.HasKey(x => x.Id);

builder.Property(x => x.Name)
       .HasMaxLength(100)
       .IsRequired();

builder.HasIndex(x => x.Email)
       .IsUnique();
```

---

# Recommended Approach for Your Projects

Since you are building:

* Smart Healthcare System
* Authentication Module with JWT
* CQRS + MediatR
* Clean Architecture
* ASP.NET Core Web APIs

A common enterprise approach is:

### Validation

Use **FluentValidation**

```csharp
RegisterCommandValidator
LoginCommandValidator
CreateDoctorCommandValidator
```

### Database Configuration

Use **EF Core Fluent API**

```csharp
DoctorConfiguration
HospitalConfiguration
AppointmentConfiguration
```

### Avoid Excessive Data Annotations

Keep entities clean:

```csharp
public class Doctor
{
    public Guid Id { get; set; }
    public string FullName { get; set; }
}
```

Configure everything in separate configuration classes:

```csharp
public class DoctorConfiguration
    : IEntityTypeConfiguration<Doctor>
{
    public void Configure(EntityTypeBuilder<Doctor> builder)
    {
        builder.Property(x => x.FullName)
               .HasMaxLength(200)
               .IsRequired();
    }
}
```

This approach scales much better as the application grows.

# Interview Question

**Can Fluent API override Data Annotations?**

✅ Yes.

If both are configured for the same property, **Fluent API takes precedence** over Data Annotations.

Example:

```csharp
[MaxLength(50)]
public string Name { get; set; }
```

```csharp
builder.Property(x => x.Name)
       .HasMaxLength(100);
```

Result:

```sql
nvarchar(100)
```

because Fluent API overrides the Data Annotation configuration.
