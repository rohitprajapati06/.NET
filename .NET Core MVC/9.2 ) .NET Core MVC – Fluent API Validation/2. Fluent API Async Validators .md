# Fluent API Async Validators in ASP.NET Core MVC

When developers talk about **Fluent API Async Validators** in ASP.NET Core MVC, they are usually referring to **FluentValidation's asynchronous validation rules**, not Entity Framework Fluent API.

Async validators are useful when validation requires:

* Database access
* API calls
* External service checks
* File existence checks
* Email uniqueness checks
* Username availability checks

---

# Why Use Async Validation?

Consider a registration form:

```csharp
public class RegisterDto
{
    public string Email { get; set; }
}
```

Before creating a user, you want to check whether the email already exists in the database.

❌ Synchronous validation:

```csharp
var exists = _context.Users.Any(x => x.Email == email);
```

This blocks the thread.

✅ Asynchronous validation:

```csharp
var exists = await _context.Users.AnyAsync(x => x.Email == email);
```

This improves scalability and performance.

# Basic Async Validator

## DTO

```csharp
public class RegisterDto
{
    public string Email { get; set; }
}
```

---

## Validator

```csharp
using FluentValidation;

public class RegisterDtoValidator
    : AbstractValidator<RegisterDto>
{
    private readonly ApplicationDbContext _context;

    public RegisterDtoValidator(
        ApplicationDbContext context)
    {
        _context = context;

        RuleFor(x => x.Email)
            .NotEmpty()
            .EmailAddress()
            .MustAsync(BeUniqueEmail)
            .WithMessage("Email already exists");
    }

    private async Task<bool> BeUniqueEmail(
        string email,
        CancellationToken cancellationToken)
    {
        return !await _context.Users
            .AnyAsync(x => x.Email == email,
                cancellationToken);
    }
}
```

# MustAsync()

`MustAsync()` is the most commonly used async validation method.

### Syntax

```csharp
RuleFor(x => x.Property)
    .MustAsync(async (value, cancellation) =>
    {
        // async validation
        return true;
    });
```

# Example: Username Availability

```csharp
RuleFor(x => x.UserName)
    .MustAsync(async (username, cancellation) =>
    {
        return !await _context.Users
            .AnyAsync(
                x => x.UserName == username,
                cancellation);
    })
    .WithMessage("Username already taken");
```

# Example: Validate Against External API

Suppose you want to verify whether a postal code exists.

```csharp
public class AddressValidator
    : AbstractValidator<AddressDto>
{
    private readonly HttpClient _httpClient;

    public AddressValidator(HttpClient httpClient)
    {
        _httpClient = httpClient;

        RuleFor(x => x.PostalCode)
            .MustAsync(IsValidPostalCode)
            .WithMessage("Invalid postal code");
    }

    private async Task<bool> IsValidPostalCode(
        string postalCode,
        CancellationToken cancellationToken)
    {
        var response = await _httpClient.GetAsync(
            $"api/postal/{postalCode}",
            cancellationToken);

        return response.IsSuccessStatusCode;
    }
}
```

# CustomAsync()

For complex validation logic.

```csharp
RuleFor(x => x.Email)
    .CustomAsync(async (
        email,
        context,
        cancellationToken) =>
    {
        bool exists = await _context.Users
            .AnyAsync(
                x => x.Email == email,
                cancellationToken);

        if (exists)
        {
            context.AddFailure(
                "Email",
                "Email already exists");
        }
    });
```

# Async Validation with Multiple Properties

Suppose FirstName + LastName combination must be unique.

```csharp
RuleFor(x => x)
    .MustAsync(async (
        model,
        cancellationToken) =>
    {
        return !await _context.Users.AnyAsync(
            x =>
                x.FirstName == model.FirstName &&
                x.LastName == model.LastName,
            cancellationToken);
    })
    .WithMessage("User already exists");
```

# Example: Product Validation

## DTO

```csharp
public class ProductCreateDto
{
    public string Name { get; set; }
}
```

---

## Validator

```csharp
public class ProductCreateDtoValidator
    : AbstractValidator<ProductCreateDto>
{
    private readonly ApplicationDbContext _context;

    public ProductCreateDtoValidator(
        ApplicationDbContext context)
    {
        _context = context;

        RuleFor(x => x.Name)
            .NotEmpty()
            .MaximumLength(100)
            .MustAsync(ProductNameAvailable)
            .WithMessage(
                "Product name already exists");
    }

    private async Task<bool> ProductNameAvailable(
        string productName,
        CancellationToken cancellationToken)
    {
        return !await _context.Products
            .AnyAsync(
                x => x.Name == productName,
                cancellationToken);
    }
}
```

# Async Validation in CQRS + MediatR

This is the most common enterprise approach.

### Command

```csharp
public class RegisterCommand
{
    public string Email { get; set; }
}
```

### Validator

```csharp
public class RegisterCommandValidator
    : AbstractValidator<RegisterCommand>
{
    public RegisterCommandValidator(
        ApplicationDbContext context)
    {
        RuleFor(x => x.Email)
            .MustAsync(async (
                email,
                cancellation) =>
            {
                return !await context.Users
                    .AnyAsync(
                        x => x.Email == email,
                        cancellation);
            });
    }
}
```

When combined with a Validation Pipeline Behavior, validation occurs before the handler executes.

This fits well with your Smart Healthcare and authentication projects because you're already implementing:

* CQRS
* MediatR
* Validation Pipeline Behavior
* Clean Architecture

# Best Practices

### Use Async for Database Checks

✅ Good

```csharp
AnyAsync()
FirstOrDefaultAsync()
CountAsync()
```

❌ Avoid

```csharp
Any()
FirstOrDefault()
Count()
```

inside validators.

---

### Pass CancellationToken

✅

```csharp
AnyAsync(
    x => x.Email == email,
    cancellationToken);
```

---

### Keep Validators Focused

Validator:

```csharp
Check email uniqueness
```

Handler:

```csharp
Create user
Generate token
Send email
```

Do not mix business logic into validators.

# Interview Questions

### What is `MustAsync()` in FluentValidation?

`MustAsync()` allows asynchronous validation logic such as database lookups and external API calls.

### When should async validators be used?

When validation requires:

* Database queries
* API requests
* External service communication
* I/O operations

### Difference between `Must()` and `MustAsync()`?

| Must()           | MustAsync()      |
| ---------------- | ---------------- |
| Synchronous      | Asynchronous     |
| No await         | Supports await   |
| CPU-bound checks | I/O-bound checks |

### Which EF Core methods are commonly used in async validators?

```csharp
AnyAsync()
FirstOrDefaultAsync()
CountAsync()
SingleOrDefaultAsync()
```

### Why are async validators important?

They prevent thread blocking, improve scalability, and are better suited for web applications that perform database or external service checks.
